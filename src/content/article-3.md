# 🔧 Claude Code 多种工作模式与高阶 Prompt 技巧
## — 从"能用"到"好用"的进阶指南

> **阅读时间**: 12 min  
> **目标读者**: 已掌握 Claude Code 基础操作，希望进阶使用的开发者  
> **版本要求**: Claude Code v1.0+

## 一、Claude Code 的三种工作模式

Claude Code 并非只有一种工作方式。深入了解其三种核心模式，能让你在不同场景下做出最优选择。

| 模式 | 触发方式 | 适用场景 | 自动化程度 |
|------|---------|----------|:----------:|
| **自由对话模式** | 直接输入需求 | 探索性任务、头脑风暴 | 低 |
| **命令模式** | `/command` | 精确操作、文件读写 | 中 |
| **Agent 自主模式** | `/agent` 或复杂任务自动触发 | 端到端项目开发 | 高 |

### 1.1 自由对话模式（默认）

这是最常用的模式，直接像和同事聊天一样描述需求：

```
我: 帮我看看这个项目的 package.json，有哪些依赖可以升级？

Claude: （分析 package.json，列出可升级的依赖和版本）
```

**最佳场景**：
- 询问代码逻辑
- 讨论技术方案
- 代码审查和评审
- 学习新概念

### 1.2 命令模式（`/`）

Claude Code 提供了一组强大的内置命令：

| 命令 | 作用 | 示例 |
|------|------|------|
| `/help` | 查看所有命令 | `/help` |
| `/clear` | 清除对话历史 | `/clear` |
| `/status` | 查看当前会话状态 | `/status` |
| `/diff` | 查看当前未提交的变更 | `/diff` |
| `/undo` | 撤销最近一次修改 | `/undo` |
| `/bug` | 启动 Bug 修复工作流 | `/bug 页面白屏` |
| `/review` | 审查代码变更 | `/review` |
| `/init` | 初始化 CLAUDE.md | `/init` |
| `/effort` | 控制思考深度（low/medium/high/xhigh） | `/effort xhigh` |
| `/goal` | 目标驱动循环，直至完成 | `/goal 实现完整登录系统` |

**进阶用法示例**：

```bash
# 查看所有未暂存的变更
/diff

# 启动结构化 Bug 报告
/bug 用户点击登录按钮后页面无响应，控制台无报错

# 一次性审查所有变更
/review

# 撤销最近的文件修改
/undo

# 设置最大思考力度
/effort xhigh
```

### 1.3 Agent 自主模式

当 Claude Code 认为任务足够复杂时，会自动进入 Agent 模式。你也可以通过描述明确触发。

**自动触发条件**：
- 涉及多个文件操作
- 需要执行终端命令
- 需要安装依赖
- 需要创建完整项目结构

**手动触发示例**：

```
我: /agent 帮我重构 src/utils/helper.js，将其拆分为多个按功能划分的模块文件

Claude Code (Agent 模式):
  📋 分析 helper.js 的导出函数
  📁 创建 src/utils/format.js
  📁 创建 src/utils/validate.js
  📁 创建 src/utils/api.js
  🔄 更新所有引用文件
  ✅ 重构完成
```

## 二、高阶 Prompt 工程技巧

### 2.1 结构化 Prompt 模板

一个好的 Prompt 应该包含以下要素：

```
[角色设定] × [任务描述] × [约束条件] × [输出格式] × [示例]

示例：
"你是一个资深 React 架构师。
帮我设计一个可复用的数据表格组件。
约束：使用 TypeScript，支持排序和筛选。
输出格式：先给类型定义，再给组件实现。"
```

### 2.2 项目级 Prompt：CLAUDE.md 的高级用法

CLAUDE.md 是真正的"项目大脑"——它定义了 AI 理解项目的上下文基线：

```markdown
# CLAUDE.md — ATM Blog

## 项目概况
这是个人博客项目，使用 React + Vite + Tailwind CSS + React Router

## 代码规范
- 组件统一使用 .jsx 扩展名
- 不要使用 TypeScript
- CSS 样式全部使用 Tailwind 类名
- 文件命名采用 kebab-case

## 架构约束
- 页面组件放在 src/pages/ 下
- 公共组件放在 src/components/ 下
- 数据文件放在 src/data/ 下
- 内容文件放在 src/content/ 下

## 安全规则
- 不经询问不要执行 npx create-* 命令
- 不经确认不要删除 package.json 中的依赖
- 任何数据库操作前必须让我确认
```

### 2.3 任务拆分与上下文控制

**黄金法则：单次 Prompt 只做一件事**

```markdown
❌ "帮我做一个博客系统，包括文章列表、详情页、
    管理后台、评论功能..."

✅ "帮我创建博客的文章列表组件"
✅ "给文章列表添加搜索和筛选功能"
✅ "创建文章详情页，支持 Markdown 渲染"
```

**上下文注入技巧**：

```markdown
# 引用文件的两种方式

方式1：在 Prompt 中直接提及
"请修改 @src/components/ArticleCard.jsx，使其支持暗色模式"

方式2：利用已有上下文
"我刚刚打开的 ArticleCard 组件中，card-title 的样式太浅了，
请加深颜色让它在亮色模式下更清晰"
```

### 2.4 学会"教"AI 理解你的偏好

AI 不是读心术，需要你明确指导。一个实用技巧是**纠正 + 记录**：

```
我: 生成的按钮太大了，高度设成 40px 就好
Claude: （修改）
我: 好的。请把这个偏好记录在 CLAUDE.md 中：
     "按钮高度统一为 40px，圆角 8px"
```

这样下次 Claude Code 就会默认使用这些偏好。

```markdown
你是一个 FastAPI 后端开发者。
请帮我创建一个用户管理 API，包含以下端点：
- POST /api/users/register  注册
- POST /api/users/login     登录
```

### 场景 2：调试复杂 Bug

```markdown
/bug

Bug 描述：
当用户在暗色模式下切换页面时，页面会短暂闪烁白色背景。

已排查：
- 不是路由加载延迟
- body 背景色已设置为 dark
- 只在 React Router 切换时出现

怀疑是：
- Tailwind 的 dark: 类在初始渲染时未正确应用
- 或者 CSS 加载顺序问题

请帮我分析 root cause 并修复。
```

### 场景 3：代码审查

```markdown
/review

1. 是否有未处理的异步错误
2. 是否有内存泄漏风险（useEffect 未清理）
3. 是否有 XSS 漏洞（用户输入未转义）
4. 是否有性能问题（不必要的重渲染）
5. 是否符合项目的命名规范
```

## 四、高级工作模式

### 4.1 `/effort`——思考力度的精细化控制

`/effort` 命令允许你**显式控制 Claude Code 在当前任务上投入的"思考深度"**。它直接映射到底层 API 的 **thinking budget（扩展思考预算）**：力度越高，分配给推理的 Token 越多，模型越倾向于"先想清楚再回答"。官方共有 **4 个命名档位**：

| 力度级别 | 命令 | 是否触发思考 | 适用场景 |
|:--------:|------|:----------:|---------|
| **low** | `/effort low` | ⚠️ 简单问题常跳过 | 高吞吐、对延迟敏感的流水线任务、批量补全 |
| **medium** | `/effort medium` | 视情况触发 | 常规代码补全、小范围改动、答疑 |
| **high** | `/effort high` | ✅ 几乎总是思考 | **默认档位**，日常编码、调试、重构 |
| **xhigh** | `/effort xhigh` | ✅ 几乎总是思考 | 架构设计、安全审计、性能优化、跨文件深度推理（部分界面也写作 `max`） |

> 💡 **关于默认值**：在 Fable 5、Opus 4.8 / 4.6、Sonnet 4.6 上默认就是 `high`；而 **Opus 4.7 默认直接是 `xhigh`**。如果你没显式设置，用的就是模型各自的默认档。

**关键认知**：`high` 和 `xhigh` 下 Claude 几乎总会先思考；`low` / `medium` 对简单问题会**主动跳过思考**以节省 Token 和延迟——所以低档并不代表"质量差"，而是"该想的时候想，不该想的时候直接答"。更有意思的是，高档位反而可能因为"想明白了"而**提前结束**推理，低档位则倾向于"问到哪做到哪"就停下。

**使用示例**：

```bash
# 快速查看文件内容——用最低力度
/effort low
帮我看看这个配置文件有哪些参数

# 写一个复杂的数据处理函数——默认力度即可
/effort high
帮我实现一个多线程爬虫，需要处理反爬和限流

# 做架构评审——最高力度
/effort xhigh
审查整个项目的安全性，列出所有风险点
```

> 💡 **选择建议**：日常开发用默认的 `high` 即可。仅在遇到复杂 Bug、关键架构决策或安全 / 性能审计时，才手动提到 `xhigh`——更高的力度意味着更大的 thinking budget、更多的 Token 消耗和更长的响应时间。

### 4.2 ULTRACODE——并行计算，处理超大任务

ULTRACODE 是 Claude Code 中最强大的模式，它的本质是激活名为 **Dynamic Workflows** 的并行计算能力。开启后，Claude 会将复杂任务拆解为多个子任务，分发到**数百个并行子代理（subagents）** 中同时处理，最后将所有结果合并输出。

#### 技术基础

ULTRACODE 并非全新的 API 级别，而是两项现有技术的组合：

- **`xhigh` effort**：Anthropic 推荐用于长周期编程任务的**最高思考深度**，让主代理能进行真正的任务规划和推理
- **会话中途系统消息 (Mid-conversation system messages)**：关键的 API 功能，允许主代理在任务进行中获得启动和管理子代理的新权限

```
用户输入大型任务
    │
    ▼
┌─────────────────────────────────────────────┐
│    主代理（Orchestrator）                      │
│    xhigh effort 进行任务规划与拆解             │
│    通过 Mid-conversation system messages      │
│    动态分配子代理                              │
└────┬──────┬──────┬──────┬──────┬──────┬──────┘
     │      │      │      │      │      │
     ▼      ▼      ▼      ▼      ▼      ▼
  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐
  │Agent│ │Agent│ │Agent│ │Agent│ │Agent│ │... │
  │  #1 │ │  #2 │ │  #3 │ │  #4 │ │  #N │ │     │
  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘ └────┘
     并行研究 → 数百个子 Agent 同时工作
     │       │       │       │       │
     └───────┴───────┴───────┴───────┘
                    │
                    ▼
          ┌──────────────────┐
          │  结果合并与综合输出  │
          └──────────────────┘
```

#### 适用场景

| 场景 | 说明 |
|------|------|
| **代码库级安全审计** | 并行扫描所有文件找安全漏洞 |
| **大规模迁移与现代化改造** | 将整个代码库从一种语言迁移到另一种（曾有案例在 **11 天内将 Bun 从 Zig 迁移到 Rust**） |
| **性能优化审计** | 同时对多个模块进行性能分析 |
| **大型测试矩阵运行** | 并行运行和修复大量测试用例 |
| **全仓库代码重构** | 跨成百上千个文件的系统级重构 |

#### 可用性说明

> ⚠️ ULTRACODE 目前在 **Claude Code CLI、桌面版和 VS Code 扩展**中作为 **"研究预览"（Research Preview）** 提供，面向 **Max、Team 和部分 Enterprise 计划**用户，需要管理员许可。此模式会消耗**大量 Token 资源**，请按需启用。

### 4.3 `/goal`——设定终点，全自动工作

`/goal` 模式的核心是让 Claude **自主迭代，直到达成预设目标**。你只需要定义一次成功标准，Claude 便会自动工作，完成任务后交还控制权，大幅减少人工干预。

#### 工作流程

```
① 设定目标
   └→ 在聊天中启动 /goal 命令，附上清晰、可验证的完成条件

② Claude 自动工作
   └→ 多轮自主行动，每轮都向目标靠近
   └→ 每轮完成后将会话文本发送给评估模型

③ 轻量级评估模型验证
   └→ 默认使用 Haiku（快速且成本低廉的模型）
   └→ 验证目标是否达成
       ├→ 未达成 → 自动开始下一轮迭代
       └→ 已达成 → 停止工作，交还控制权

④ 安全机制
   └─→ 可在命令中添加 "或运行 N 轮后停止" 防止无限循环
```

```
                     /goal "让测试套件全部通过"
                              │
                              ▼
                    ┌─────────────────────┐
                    │  Claude 自主工作      │
                    │  每轮 → 修改代码      │
                    │      → 运行测试      │
                    └─────────┬───────────┘
                              │
                              ▼
                    ┌─────────────────────┐
                    │  Haiku 评估模型      │
                    │  (轻量级、低成本)     │
                    └─────────┬───────────┘
                              │
                    ┌─────────┴──────────┐
                    ▼                    ▼
               未达成目标            已达成目标
                    │                    │
                    ▼                    ▼
              继续下一轮             交还控制权
               (最多 N 轮)           ✅ 完成
```

#### 实战示例

```bash
# 让测试套件从失败变为通过
/goal 修复所有失败的测试用例，让测试套件全部通过后停止。最多运行10轮。

# 跨多文件代码重构
/goal 将 src/utils/ 下所有工具函数从 CommonJS 迁移到 ESM 模块语法。
确保每个文件都更新了 import/export 语句，并且 npm run test 全部通过。
```

### 4.4 `/goal` vs ULTRACODE 对比与组合使用

#### 直观对比

| 特性 | `/goal` 模式 | ULTRACODE 模式 |
|------|:-----------:|:--------------:|
| **核心工作流** | 串行迭代，自主循环 | 并行处理，多代理协作 |
| **核心优势** | 减少人工干预，解放双手 | 大幅缩短大型任务的耗时 |
| **适用场景** | 流程固定、重复性高、有明确完成标志的任务 | 任务结构复杂、可拆解、并行度高的任务 |
| **资源消耗** | 较低 | 较高 |
| **评估机制** | Haiku 轻量模型验证 | Orchestrator 调度汇总 |

#### 选用建议

| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| **有明确的完成标准** | `/goal` | 自主迭代直到目标达成 |
| **庞大且可拆分的任务** | ULTRACODE | 并行处理，大幅缩短时间 |
| **修复所有测试失败** | `/goal` | 明确的目标条件 |
| **全仓库安全审计** | ULTRACODE | 可并行扫描所有模块 |
| **代码库跨语言迁移** | ULTRACODE | 大规模并行转换 |

#### 组合使用——效率与精度的平衡

一个高效的实践是 **组合使用两种模式**：

```
Step 1: ULTRACODE 并行处理
        ├── 子任务 A: 用户模块重构
        ├── 子任务 B: 订单模块重构    ← 数百个子 Agent 并行工作
        ├── 子任务 C: 支付模块重构
        └── 子任务 D: 通知模块重构

Step 2: 对每个子任务使用 /goal 精细打磨
        ├── /goal "让用户模块所有测试通过"
        ├── /goal "让订单模块所有测试通过"  ← 串行迭代至完美
        ├── /goal "让支付模块所有测试通过"
        └── /goal "让通知模块所有测试通过"
```

先用 ULTRACODE 并行处理项目的多个核心模块，再在某个具体子任务上使用 `/goal` 模式让 AI 自动迭代、精细打磨，实现**效率与精度的平衡**。

### 4.5 实战 Prompt 模板

#### `/goal` 模式 Prompt

一个可靠的 `/goal` 提示词，应当把**目标、约束、循环条件、终止条件**四要素拆开写清楚，避免模型自由发挥导致跑偏或陷入无限循环：

```
/goal

【目标 Goal】（必须可机器验证的成功标准）
让 src/ 下所有单元测试通过：运行 `npm test` 退出码为 0，且输出 "X passing, 0 failing"。

【约束 Constraints】（绝不能违反的红线）
- 只允许修改 src/ 目录下的 .js / .ts 文件，禁止改动任何 *.test.* / *.spec.* 测试文件
- 禁止删除或用 it.skip / xit 跳过任何现有测试用例
- 不得引入新的第三方依赖；如确有必要，必须先停下来向我说明并等待确认
- 每次改动必须保证 `npm run lint` 不新增任何报错

【循环条件 Loop Conditions】（每一轮做什么、如何逼近目标）
- 每轮开头先运行 `npm test`，获取当前失败用例清单
- 每轮只针对**一个**根因修复（不要一次性铺开改多处），优先修失败数最多的模块
- 修复后立即重跑该模块相关测试，确认这一处确已修复、且未引入新的失败

【终止条件 Termination Conditions】（满足后立即停止并交还控制权）
- 成功：连续 2 轮 `npm test` 输出 "0 failing" → 停止，输出本轮改动总结
- 上限：累计运行满 10 轮仍未通过 → 停止，输出剩余失败用例的 文件路径、行号、报错信息
- 熔断：某一轮引入的新失败数 > 本轮修复数 → 回滚本轮改动并停止，报告异常情况
```

**四要素说明**：

| 要素 | 作用 | 常见错误 |
|------|------|---------|
| **目标 Goal** | 给评估模型一个**可机器验证**的判定标准（命令输出、退出码、字符串匹配） | 写成"改进性能"这种模糊目标 → 应改为"使首页 FCP ≤ 1.2s" |
| **约束 Constraints** | 划定不可逾越的红线，防止模型为了"达成目标"而作弊（删测试、加 skip、乱引依赖） | 只说目标不说约束 → 模型可能删掉失败测试来"通过" |
| **循环条件 Loop Conditions** | 约束每一轮的行为粒度，避免一次性大改难以定位 | 让模型"自由发挥" → 改动越积越多，一旦出错无法回溯 |
| **终止条件 Termination Conditions** | 明确**何时算成功、何时必须停**，并加一个硬性轮数上限 + 熔断条件 | 不设上限 → 可能陷入无限循环持续烧 Token |

#### ULTRACODE 模式 Prompt

```
请使用 ultracode 模式处理以下任务：

【任务】对整个 `backend/` 目录下的所有 Python 文件进行安全检查。

【分解方式】
- 按子目录分组：`api/`, `services/`, `models/`, `utils/`
- 每个子目录单独成为一个子任务，并行执行。
- 每个子任务需要检查的问题类型：
  * SQL 注入风险（字符串拼接 SQL）
  * 使用 `eval()` 或 `exec()`
  * 硬编码的密钥/密码
  * 未验证的用户输入直接传递到 shell 命令

【最终输出】
- 合并所有子任务的结果，生成一个 `security_report.md`。
- 报告中按风险等级（高/中/低）排序，每个问题附带文件路径、行号和修复建议。

【额外要求】
- 请不要修改任何源代码，只做只读分析。
- 如果某个子目录不存在或无法分析，跳过并注明原因。
```

**使用要点**：
- 明确要求使用 ultracode 模式（部分界面需先开启该模式，然后在对话中描述任务分解方式）
- 指出**如何拆分**子任务（按目录、按文件类型、按功能模块等）
- 规定**并行输出的合并格式**，让 AI 知道最终需要哪种结构化结果

#### ULTRACODE 模式 Prompt（二）：技术调研（RESEARCH）

ULTRACODE 同样适合**并行调研**类任务——把一个开放性研究问题拆给多个子 Agent，各自独立查证，再汇总成一份带来源的报告：

```
请使用 ultracode 模式完成一次技术选型调研：

【调研主题】
对比 2025 年主流 React 状态管理方案，产出一份可直接决策的选型报告。

【分解方式】（每个子任务交给一个并行子 Agent 独立调研）
- 子任务 A：Redux Toolkit
- 子任务 B：Zustand
- 子任务 C：Jotai
- 子任务 D：MobX
- 子任务 E：Valtio
- 子任务 F：React Context + useReducer（无第三方库的基线方案）

每个子任务必须按统一字段输出，方便后续横向对比：
  方案名 / 包体积 / 核心 API / 学习曲线 / 适用场景 / 不适用场景 /
  性能特征（大列表渲染、高频更新）/ SSR 支持 / 维护活跃度 /
  Star 与周下载量 / 官方文档链接

【合并与综合】
1. 汇总所有子任务结果为一张横向对比表（方案 × 维度）。
2. 按典型场景给出推荐：中小型应用、大型应用、强 SSR、高频实时更新。
3. 任何结论都必须附来源链接；无法证实的结论标注「待核实」，不得编造数据。

【约束】
- 只做调研，不修改仓库中的任何代码。
- 数据优先取自官方文档与 GitHub / npm，二手博客需注明来源。
```

**使用要点**：
- RESEARCH 任务的关键是**给每个子 Agent 规定统一的输出 schema（字段）**，否则汇总时格式参差不齐、无法横向对比
- 明确要求**带来源 + 标注「待核实」**，这是降低调研类任务"幻觉"最有效的约束
- 规定**合并形式**（对比表 + 场景推荐），让最终产出可直接用于决策

#### 使用小技巧

| 模式 | 提示词关键点 | 常见错误 |
|------|-------------|---------|
| `/goal` | 完成条件必须**可验证**（如"运行某个命令后输出 OK"） | 目标太模糊——"改进性能" → 应改为"使首页 FCP ≤ 1.2 秒" |
| ULTRACODE | 必须指明**如何拆分**以及**如何合并** | 只说"请并行处理所有文件" → 缺少合并逻辑，结果杂乱 |

```markdown
安全配置建议：

{
  "permissions": {
    "file_ops": true,        // ✅ 允许文件操作
    "shell_commands": true,  // ✅ 允许终端命令
    "network_access": false, // ❌ 禁止网络请求
    "dangerous_commands": [] // ❌ 禁止危险命令
  }
}
```

**企业项目**：

```markdown

```

## 六、常见误区

| 误区 | 正确做法 |

## 七、总结

Claude Code 的三种基础模式——自由对话、命令模式、Agent 自主模式——配合新引入的 `/effort`、ULTRACODE、`/goal` 等高级工作模式，覆盖了从简单问答到大规模并行开发的全部场景。配合 CLAUDE.md 和高阶 Prompt 技巧，你能将开发效率提升 **5-10 倍**。

关键要点：

```
✅ 根据任务复杂度选择模式——简单对话、中等用 Agent、复杂用 /goal
✅ 用 /effort 控制思考深度——日常用 high（默认），复杂用 xhigh
✅ 大规模并行任务用 ULTRACODE——全仓库审计、跨文件重构
✅ /goal 实现端到端自动化——明确目标后放手让 AI 循环执行
✅ CLAUDE.md 是必须配置的"项目大脑"
✅ 好的 Prompt = 角色 + 任务 + 约束 + 格式
✅ 始终审查 AI 生成的代码
```

> **一句话总结：** Claude Code 的三种工作模式配合结构化的 Prompt 工程，能让 AI 从"能写代码"进化到"会写符合项目规范的代码"——关键是学会"正确地提问"。
